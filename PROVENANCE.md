# VaultSend — crypto bundle provenance

The only third-party code in `VaultSend_offline.html` is a single inline
`<script id="vs-bundle">` IIFE: a tree-shaken bundle of the `age` encryption
stack. This document records how to prove that those exact bytes were built from
genuine, pinned upstream sources — and the result of doing so.

## What is pinned

| Package            | Version | Author        | License |
|--------------------|---------|---------------|---------|
| age-encryption     | 0.2.4   | F. Valsorda   | MIT |
| @noble/ciphers     | 1.3.0   | P. Miller     | MIT |
| @noble/curves      | 1.9.7   | P. Miller     | MIT |
| @noble/hashes      | 1.8.0   | P. Miller     | MIT |
| @scure/base        | 1.2.6   | P. Miller     | MIT |
| fflate             | 0.8.2   | A. Gao        | MIT |
| esbuild (builder)  | 0.28.1  | —             | MIT |

Bundle integrity is also enforced at runtime by the page's Content-Security-
Policy, which pins the SHA-256 of the script. The browser refuses to execute any
script whose hash is not pinned.

- CSP pin (base64): `sha256-ICmTDSgHvTs31dRxgZS3fdCGun1CChgXbL3GFpYMnrI=`
- Same value (hex): `2029930d2807bd3b37d5d4718194b77dd086ba7d420a18176cbdc616960c9eb2`
- Bundle size: 77,656 bytes

## Reproduce it yourself (one command)

From the folder containing the HTML file and `verify.sh`:

```sh
bash verify.sh VaultSend_offline.html
```

This extracts the embedded bundle, rebuilds it from the pinned upstream versions
with the exact `esbuild` invocation, and compares hashes. Expected output:

```
embedded bundle sha256 : 2029930d2807bd3b37d5d4718194b77dd086ba7d420a18176cbdc616960c9eb2
rebuilt  bundle sha256 : 2029930d2807bd3b37d5d4718194b77dd086ba7d420a18176cbdc616960c9eb2
expected (pinned)      : 2029930d2807bd3b37d5d4718194b77dd086ba7d420a18176cbdc616960c9eb2
RESULT: ✅ PASS — embedded bundle is byte-identical to the pinned upstream rebuild
```

### Manual equivalent

```sh
npm i age-encryption@0.2.4 fflate@0.8.2 esbuild@0.28.1 \
      @noble/ciphers@1.3.0 @noble/curves@1.9.7 @noble/hashes@1.8.0 @scure/base@1.2.6
# entry.js:
#   import {generateIdentity,identityToRecipient,Encrypter,Decrypter,armor} from "age-encryption";
#   import {zipSync} from "fflate";
#   globalThis.__VS_AGE__={generateIdentity,identityToRecipient,Encrypter,Decrypter,armor};
#   globalThis.__VS_FFLATE__={zipSync};
npx esbuild entry.js --bundle --format=iife --platform=browser \
  --target=es2020 --minify --legal-comments=none --outfile=vendor.min.js
sha256sum vendor.min.js   # -> 2029930d…960c9eb2
```

## Verification status

This recipe was executed and the rebuild matched the embedded bundle
**byte-for-byte** (identical SHA-256, 77,656 bytes). Provenance is therefore
reproducible, not merely asserted.

## Two independent guarantees in the app

1. **Menu → Verify integrity** — recomputes the SHA-256 of both inline scripts
   running in your browser and confirms they match the CSP pins (and that the
   bundle equals this documented hash). Tool-free.
2. **Menu → Run crypto self-test** — decrypts ciphertexts and reproduces an
   identity→recipient mapping that were generated by the genuine upstream
   `age-encryption@0.2.4`, plus a live round-trip and a 1-bit tamper-rejection
   check. This proves the embedded X25519 / HKDF / ChaCha20-Poly1305 / scrypt
   behave exactly like upstream — catching a weakened bundle even without a
   rebuild.

## What this does and does not prove

- **Proven:** the embedded bundle is byte-identical to a clean build of the
  pinned upstream releases, and the running code implements standard age crypto
  that interoperates with upstream test vectors.
- **Not proven by these tools alone:** that the *pinned upstream releases
  themselves* are free of supply-chain compromise. For that, rely on the noble/
  age maintainers' own signing/audits and npm provenance. Pinning + reproducible
  build means you only have to trust those named releases — nothing this file or
  any CDN could silently swap.
