# VaultSend

A small, auditable desktop app for encrypting files and text to people you
choose, using the [age](https://github.com/str4d/rage) encryption format.

It is built so that one person can read the whole thing in an afternoon. There
are two parts and a hard line between them:

- **`backend/`** — a tiny Rust command-line tool. It is the **only** component
  that ever touches your private key. All cryptography is delegated to the `age`
  crate; the backend itself contains no hand-written crypto, just plumbing
  (~235 lines, one file).
- **`frontend/`** — a GTK 4 / libadwaita app in Python. It draws the window and
  keeps your contact list. It **never sees your private key** — when a passphrase
  is needed it hands it to the backend over a private pipe and forgets it.

## Security model in one paragraph

Your private key is generated by the backend, encrypted with a passphrase you
choose (age's scrypt passphrase encryption), and written to disk as
`identity.age` with `0600` permissions. Your **public** key is cached next to it
in the clear, because a public key is not a secret — this is what lets you
encrypt and look up your own address without typing a passphrase. The passphrase
is therefore only ever needed to **decrypt**. Encrypting to several recipients
does not weaken anything: age encrypts the body once under a single random key
and just wraps a copy of that key to each recipient separately.

## Build

The backend is the only thing that needs compiling:

```sh
cd backend
cargo build --release
```

This produces `backend/target/release/vaultsend-backend`. A `Cargo.lock` is
checked in and pins the dependency tree to versions that build on **Rust 1.75+**
(age itself targets Rust 1.65; a few of its localization dependencies published
newer releases that require a newer compiler, so they are pinned down). `age` is
the sole cryptographic dependency.

## Run

The frontend needs GTK 4 (≥ 4.10) and libadwaita (≥ 1.5) with their Python
bindings. **Install these from your distribution, not from pip** — PyGObject
from pip will not find your system GTK.

Debian / Ubuntu:

```sh
sudo apt install python3-gi gir1.2-gtk-4.0 gir1.2-adw-1
```

Fedora:

```sh
sudo dnf install python3-gobject gtk4 libadwaita
```

Arch:

```sh
sudo pacman -S python-gobject gtk4 libadwaita
```

Then:

```sh
make run        # or: python3 frontend/app.py
```

On first launch you'll be asked to choose a passphrase; that creates your key.
The frontend finds the backend by, in order: the `VAULTSEND_BACKEND`
environment variable, the sibling `backend/target/release/` build, then `PATH`.

## Where things are stored

| What | Path |
|------|------|
| Encrypted private key (`0600`) | `~/.local/share/vaultsend/identity.age` |
| Public key (clear) | `~/.local/share/vaultsend/pubkey` |
| Contacts (plain, exportable JSON) | `~/.config/vaultsend/contacts.json` |

(`XDG_DATA_HOME` / `XDG_CONFIG_HOME` are honored if set.) Contacts are just a
list of `{"name", "pubkey"}` objects — copy the file to back it up or move it.

## Using it

- **Encrypt text:** type or paste a message, click **Encrypt**, choose
  recipients (you are always included so you can re-read it), and the box is
  replaced with armored text you can copy and send.
- **Decrypt text:** paste the armored text, click **Decrypt**, enter your
  passphrase.
- **Encrypt / decrypt a file:** drop it on the window or use **Open file…**,
  pick the operation, and choose where to save the result.
- **Contacts:** add someone by pasting their `age1…` public key. Click a contact
  to copy their key.
- **Your public key** lives in the menu (⋮) — share it so others can write to you.

## Backend command-line interface

The backend is usable on its own and easy to audit in isolation:

```
vaultsend-backend keygen  --pass-fd N
vaultsend-backend pubkey
vaultsend-backend encrypt -r age1.. [-r age1..]... [--armor] [--in PATH] [--out PATH]
vaultsend-backend decrypt --pass-fd N [--in PATH] [--out PATH]
```

Secrets are never passed as arguments (those are visible to other processes):
the passphrase is read from the file descriptor named by `--pass-fd`. File data
flows over stdin/stdout unless `--in`/`--out` are given.

## Where to look when auditing

- `backend/src/main.rs` — the four `cmd_*` functions are the whole story;
  `cmd_encrypt` / `cmd_decrypt` are the crypto paths.
- `frontend/app.py` — `run_backend()` is the entire trust boundary: it shows
  exactly what is sent to the backend and how the passphrase is passed.

## Scope / limitations

- The backend is invoked synchronously, so encrypting a very large file will
  briefly freeze the window. Fine for typical use; kept this way for simplicity.
- One identity per user; no key fetching over a network; no plugins. By design.
